## 1. Goal

Replace Electron's `globalShortcut` (which is partial-to-broken on Wayland) with a `uiohook-napi`-based hotkey path that works uniformly on Linux X11, Linux Wayland, macOS, and Windows. Preserve all Phase 5.3 hotkey UX (rebinding, conflict detection, inc/dec semantics) â€” only the underlying OS mechanism changes.

The motivating user is a Fedora 43 + Wayland developer whose hotkeys are silently failing today.

## 2. Non-Goals

- App packaging (electron-builder configs) â€” Phase 6.1.
- Auto-update channel â€” Phase 6.4.
- Visual toast component for the hotkeys-unavailable banner â€” Phase 6.5 (uses `window.alert` for v1, matching the Phase 5.1 corrupt-config pattern).
- Per-key event throttling / repeat detection â€” uiohook fires `keydown` once per OS keydown; OS-level autorepeat surfaces as multiple events. Acceptable for v1.
- Capturing keys when the app is in the background. uiohook fires globally regardless of focus, which is the desired behavior â€” same as Electron's `globalShortcut`.

## 3. Architecture

Three boundaries:

- **`hotkey-matcher.ts`** is the only place accelerator-string parsing happens. Pure function: `parseAccelerator(s: string): AcceleratorMatcher | null`. The matcher is a `(event: UiohookKeyboardEvent) => boolean` predicate. Fully unit-testable.
- **`hotkey-host.ts`** is the only file importing `uiohook-napi`. Owns the listener lifecycle and event-loop dispatch. Public API unchanged from Phase 5.3 (`registerForTrainer`, `unregisterAll`, `syncCheatState`) plus new `shutdown` and `setInitFailureHandler`.
- **`hotkey-matcher.ts` and `hotkey-host.ts` together replace** the previous `hotkey-host.ts`'s `globalShortcut`-based registration. All `globalShortcut` imports/usages are deleted.

## 4. Accelerator Matcher

`parseAccelerator(s)` accepts Electron-format strings (the existing canonical format used by the catalog, user config, importer output, and `HotkeyCapture` widget) and returns a predicate over `UiohookKeyboardEvent`.

### 4.1 Token vocabulary

- **Modifiers** (any order, before the key): `Ctrl` / `Control`, `CommandOrControl` / `CmdOrCtrl` (translates at parse time to `Cmd` on darwin, `Ctrl` elsewhere), `Cmd` / `Command` / `Super`, `Alt` / `Option`, `Shift`.
- **Key** (last token): letters `A`â€“`Z`, digits `0`â€“`9`, function keys `F1`â€“`F24`, arrows `Up`/`Down`/`Left`/`Right`, common specials (`Space`, `Tab`, `Return`, `Backspace`, `Delete`, `Insert`, `Home`, `End`, `PageUp`, `PageDown`), punctuation (`=`, `-`, `[`, `]`, `;`, `'`, `,`, `.`, `/`, `\`, `` ` ``), numpad `num0`â€“`num9`.

### 4.2 Key code mapping

A `const KEY_TOKEN_TO_UIOHOOK: Record<string, number>` maps each supported token to the libuiohook keycode. Built from `UiohookKey` constants exported by `uiohook-napi` plus a few additions for tokens uiohook doesn't pre-name. Unknown tokens cause `parseAccelerator` to return `null`.

### 4.3 Match semantics

The matcher returns `true` iff:

1. `event.keycode` equals the parsed keycode.
2. `event.ctrlKey === expectsCtrl`, `event.altKey === expectsAlt`, `event.shiftKey === expectsShift`, `event.metaKey === expectsMeta` â€” **all four** must match exactly. `Ctrl+F4` does NOT fire when the user presses `Ctrl+Shift+F4`. This avoids overlapping accelerators (e.g., a value cheat with `toggle: F4`, `inc: F4+Up`, `dec: F4+Down`) firing the wrong slot.

### 4.4 Tests

8 tests in `hotkey-matcher.test.ts` covering: single keys, single-modifier combos, multi-modifier, `CommandOrControl` cross-platform translation, unknown token rejection, exact-modifier-match enforcement, keycode mismatch rejection, modifier-mismatch rejection.

## 5. hotkey-host

### 5.1 State

```ts
interface RegisteredCheat {
  cheatId: string;
  toggleMatcher?: AcceleratorMatcher;
  incMatcher?: AcceleratorMatcher;
  decMatcher?: AcceleratorMatcher;
}

const cheats: RegisteredCheat[] = [];
const isOn = new Map<string, boolean>();
let started = false;
let onInitFailure: ((message: string) => void) | null = null;
```

### 5.2 Public API

```ts
export function registerForTrainer(t: StarlightTrainer | null, overrides?: TrainerHotkeyOverrides): void;
export function unregisterAll(): void;
export function syncCheatState(cheatId: string, on: boolean): void;
export function shutdown(): void;
export function setInitFailureHandler(fn: (message: string) => void): void;
```

`registerForTrainer` builds matchers via `parseAccelerator`, populates `cheats`, and lazily starts the uiohook listener (`ensureStarted`). `unregisterAll` clears `cheats` but leaves the listener running (it'll just dispatch nothing). `shutdown` is called from `app.before-quit` and tears down the libuiohook thread.

### 5.3 Listener startup

```ts
function ensureStarted(): void {
  if (started) return;
  try {
    uIOhook.on('keydown', onKeyDown);
    uIOhook.start();
    started = true;
  } catch (err) {
    uIOhook.off('keydown', onKeyDown);
    onInitFailure?.(err instanceof Error ? err.message : String(err));
    // Do not re-throw â€” let registerForTrainer return normally with no live hotkeys.
  }
}
```

A failure here means hotkeys are unavailable for the rest of the app's lifetime. The renderer surfaces this once via `window.alert`.

### 5.4 Dispatch loop

```ts
async function onKeyDown(e: UiohookKeyboardEvent): Promise<void> {
  for (const reg of cheats) {
    if (reg.toggleMatcher?.(e)) {
      const next = !(isOn.get(reg.cheatId) ?? false);
      const r = await engineHost.toggleCheat(reg.cheatId, next);
      if (r.ok) {
        isOn.set(reg.cheatId, next);
        broadcast({ type: 'cheat:toggled', cheatId: reg.cheatId, on: next, cause: 'hotkey' });
      }
      continue;
    }
    if (reg.incMatcher?.(e)) {
      const r = await engineHost.incCheat(reg.cheatId);
      if (r.ok) broadcast({ type: 'hotkey:inc', cheatId: reg.cheatId });
      continue;
    }
    if (reg.decMatcher?.(e)) {
      const r = await engineHost.decCheat(reg.cheatId);
      if (r.ok) broadcast({ type: 'hotkey:dec', cheatId: reg.cheatId });
      continue;
    }
  }
}
```

- First match per slot wins; `continue` prevents one event from firing two cheats with the same accelerator (Phase 5.3 conflict detection prevents this in normal use).
- Hot path: ~60 events/sec while typing. N comparisons (N = supported cheats Ă— up-to-3 slots), each a closure call with a few field reads. No per-event allocation.

### 5.5 Shutdown

`main/index.ts`'s `before-quit` handler:

```ts
app.on('before-quit', async () => {
  hotkeyHost.shutdown();
  unregisterHotkeys();              // existing â€” clears module state
  await engineHost.detach();
});
```

`shutdown()` is idempotent: returns early if `started === false`.

## 8. Migration / Removal

- **`globalShortcut` removed** from `hotkey-host.ts` and from the `electron` import in `main/index.ts`. The existing `before-quit` line `globalShortcut.unregisterAll()` is removed (it's a no-op once `globalShortcut` is unused, but keep things clean).
- **`will-quit` handler** in `main/index.ts` (which currently calls `globalShortcut.unregisterAll()`) â€” drop the call; replace with `hotkeyHost.shutdown()` if not already covered by `before-quit`.
- **No data migration needed.** Catalog trainer files, user-config `hotkeyOverrides`, and the `HotkeyCapture` widget all already produce/consume Electron-format accelerator strings. The `hotkey-matcher` parses the same strings â€” the wire format is preserved.

## 9. Native Module Concerns

- **Install-time:** `uiohook-napi` ships prebuilt binaries via `prebuild-install` for common combos (Node 20 + Electron 32 + glibc x64). Falls back to source build via node-gyp on uncommon targets. Linux x64/glibc (CI's `ubuntu-latest`) is well-supported.
- **Externalization:** the existing `externalizeDepsPlugin()` in `electron.vite.config.ts` already handles native modules in `dependencies`. `uiohook-napi` lands in `dependencies`; no config change.
- **Electron rebuild:** for production packaging, electron-builder's postinstall step rebuilds native modules against the Electron ABI. That's Phase 6.1's concern; for `pnpm dev` and `vitest`, the Node ABI is what matters and the prebuild should load directly.
- **Linux permission:** users not in the `input` group may hit `EACCES` on libinput devices. Documented in the failure-mode alert. Most desktop distros add the logged-in user to `input` automatically.
- **macOS Accessibility:** first-run prompt. Documented in the alert.
- **Windows:** no permission prompts.

## 10. Risks

- **Native ABI breakage** between `uiohook-napi`'s shipped prebuild and the user's Node/Electron combo. Mitigation: `prebuild-install` falls back to source build; if both fail, hotkeys are unavailable but the rest of the app works via the failure handler.
- **uiohook fires for keys when the app is in the background.** This is the desired behavior â€” global hotkeys must work when the user is in the game, which is the foreground app. If a future feature wants per-app filtering, it's additive.
- **Suppression of native key delivery.** uiohook's `keydown` event is informational only; it does NOT swallow the keypress (unlike Electron `globalShortcut`, which does on some platforms). This means pressing `F4` to toggle a cheat ALSO sends `F4` to the focused application. For trainers that's typically fine (games rarely listen for F1â€“F12 outside of menus), but if a user binds a cheat to a hotkey the game uses, both will fire. We accept this for v1; spec Â§11 acknowledges as a follow-up.
- **`UiohookKey` constants stability.** uiohook-napi's exported keycodes are a fixed enum; bumping the package shouldn't break the matcher, but a major version bump should be checked.
- **Multiple windows.** All windows receive the same `cheat:toggled` / `hotkey:inc` / `hotkey:dec` broadcasts â€” matches Phase 5.3 behavior.
