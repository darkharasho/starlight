# Phase 4.5 — Auto-Detection & Value-Cheat Hotkeys Design

**Status:** Design (pre-plan)
**Predecessor:** Phase 4 (engine integration via IPC) — committed.
**Successor:** Phase 5 (catalog repo + community trainer index).

## 1. Goals

Phase 4 left three items deferred. Phase 4.5 lands all three:

1. **Library auto-detection** — surface installed games on the Library route. Steam only in 4.5; the scanner architecture is pluggable so Epic/Heroic/Lutris drop in later without re-architecture.
2. **Process auto-detection** — replace the free-text PID input on the Active Trainer route with a process picker; auto-highlight the entry whose name matches the loaded trainer's `processName`.
3. **Inc/Dec hotkeys for value cheats** — extend the existing toggle hotkeys with `inc` / `dec` accelerators that step the cheat's value up/down within `[min, max]` while the cheat is active.

## 2. Non-Goals

- Epic / Heroic / Lutris scanners (interface ships, implementations stubbed; Phase 5+).
- Catalog repo / "has trainer" badge logic (Phase 5).
- Persisting edited `processName` to disk (in-memory in 4.5; Phase 5 user-config).
- Manual library entries (user pastes an exe path) — Phase 5.
- Wayland `globalShortcut` workarounds — Phase 6.

## 3. Architecture

## 4. IPC Surface (extends `shared/ipc.ts`)

```ts
// New methods on StarlightApi
scanLibrary(): Promise<{ games: DetectedGame[] }>;
listProcesses(): Promise<{ processes: DetectedProcess[] }>;
setProcessName(names: string[]): Promise<void>;

// New StarlightEvent variants
| { type: 'process:list'; processes: DetectedProcess[] }
| { type: 'process:matched'; pid: number; name: string }
| { type: 'library:scanned'; games: DetectedGame[] }
| { type: 'hotkey:inc'; cheatId: string }
| { type: 'hotkey:dec'; cheatId: string }

// New shared types
export interface DetectedGame {
  source: 'steam';                       // future: 'epic' | 'heroic' | 'lutris'
  appId: string;
  name: string;
  installDir: string;                    // absolute path
}
export interface DetectedProcess { pid: number; name: string }
```

`scanLibrary` is request/response (called when Library route mounts and on Refresh).
`listProcesses` is request/response (called by the picker dropdown).
`process:list` and `process:matched` flow over the existing `onEvent` channel from the polling loop.

## 5. Library Auto-Detection

### 5.1 Scanner interface

```ts
export interface LibraryScanner {
  readonly source: DetectedGame['source'];
  scan(): Promise<DetectedGame[]>;       // never throws; returns [] on failure
}
```

`scanAll()` runs every registered scanner concurrently with `Promise.all` + per-scanner `.catch(() => [])`, so one scanner failing does not poison the rest. Phase 4.5 registers `SteamScanner` only; `EpicScanner`, `HeroicScanner`, `LutrisScanner` ship as classes with `scan()` returning `[]` and a `// TODO: Phase 5` comment, exercising the pluggability without shipping half-finished platform code.

### 5.2 SteamScanner

1. **Resolve Steam install root** by platform:
   - Linux: try `~/.steam/steam`, `~/.local/share/Steam`, `~/.var/app/com.valvesoftware.Steam/.steam/steam` (Flatpak); first that exists wins.
   - macOS: `~/Library/Application Support/Steam`.
   - Windows: `%PROGRAMFILES(X86)%\Steam`, fall back to reading `HKLM\Software\Valve\Steam\InstallPath` from the registry.
   - If none found → return `[]`.
2. **Parse `<root>/steamapps/libraryfolders.vdf`** with a tiny ad-hoc VDF parser (no dependency — VDF is ~30 LoC of recursive descent over `"key" "value"` and `"key" { ... }`). Yields a list of library paths.
3. **For each library path**, glob `steamapps/appmanifest_*.acf`, parse VDF, extract `appid`, `name`, `installdir`. Compose `installDir = <library>/steamapps/common/<installdir>`.
4. Return aggregated `DetectedGame[]`.

The scanner is pure (filesystem in, structured data out), so it's unit-testable with fixtures rather than requiring a real Steam install.

### 5.3 Tests

Committed fixtures under `apps/desktop/test/fixtures/steam/`:
- A `libraryfolders.vdf` listing two library paths.
- A handful of `appmanifest_*.acf` files in each library.

Tests inject the fixture root, assert the resulting `DetectedGame[]` (count, names, install paths). Cover: missing root → `[]`, malformed VDF → `[]`, library that exists but has no `appmanifest_*.acf` → `[]`.

## 6. Process Auto-Detection

### 6.1 ProcessHost

```ts
interface ProcessHostOptions {
  intervalMs: number;                    // 2000
  emit: (e: StarlightEvent) => void;
  isAttached: () => boolean;
  psList: () => Promise<{ pid: number; name: string }[]>;  // injectable for tests
}

class ProcessHost {
  start(): void;
  pause(): void;
  resume(): void;
  setTrainerProcessNames(names: string[]): void;
  clearTrainer(): void;
}
```

### 6.2 Polling lifecycle

Wired in `main/index.ts`:

- `app.on('ready')` → `processHost.start()`.
- `BrowserWindow` events: `'hide'`, `'minimize'` → `pause()`; `'show'`, `'restore'` → `resume()`. (Not `'blur'` — the user moving focus to the game itself must NOT pause polling.)
- On attach: `pause()` (the user has the PID; further polling is wasted work). On detach: `resume()`.
- `tick()` early-returns when `isAttached()` is true, as a belt-and-braces check in case the pause was missed.
- On trainer load: `setTrainerProcessNames(currentTrainer.game.processName)`.

### 6.3 Matching

Case-insensitive, evaluated in order:
1. Exact match against any `processName[]` entry.
2. Strip trailing `.exe` from both sides and retry.
3. First match wins (lowest pid).

Emits `process:matched` only on transitions (when `pid` differs from the last emitted pid). When no match, resets `lastMatchedPid` so a future re-appearance re-emits.

Every tick also emits `process:list` with the full enumeration; the picker UI consumes this.

### 6.4 listProcesses() handler

For the picker dropdown. Calls `psList()` directly — a single fresh enumeration independent of the poll loop, so the picker works even when polling is paused.

### 6.5 Tests

Inject fake `psList` and a controllable clock; assert:
- Pause stops ticks; resume restarts.
- Attach gate (`isAttached() === true` → no emit).
- `process:matched` dedup across consecutive ticks.
- `.exe`-stripping match.
- `clearTrainer()` clears `lastMatchedPid`.

## 7. Trainer `processName` Defaulting

`trainer-loader.ts` currently hard-codes `processName: ['unknown']`. Replace with:

```ts
const base = path.basename(filePath, '.CT');
processName: [base, `${base}.exe`]
```

Bare name first (Linux/Proton — Frida sees the wine-wrapped exe by basename), `.exe` second (Windows). The matcher already iterates the array.

User can override via the new "Trainer Info" disclosure on `ActiveTrainerRoute` (calls `setProcessName()` IPC). In-memory only in 4.5; Phase 5 will persist to user config.

## 8. Inc/Dec Hotkeys for Value Cheats

### 8.1 hotkey-host refactor

Phase 4 registers a single accelerator per cheat (`toggle`). Phase 4.5 generalizes to slots:

```ts
interface CheatHotkeys { toggle?: string; inc?: string; dec?: string }

registerCheat(cheatId: string, hotkeys: CheatHotkeys): void;
unregisterCheat(cheatId: string): void;
unregisterAll(): void;
```

The importer already produces `hotkeys.toggle / inc / dec` as Electron-accelerator strings, so `hotkey-host` consumes them directly. Accelerators that `globalShortcut.register` rejects are silently skipped (existing behavior, unchanged).

Each registered hotkey dispatches:
- `toggle` → `engineHost.toggleCheat(cheatId, !isActive(cheatId))` (existing).
- `inc` → `engineHost.incCheat(cheatId)` + emit `hotkey:inc`.
- `dec` → `engineHost.decCheat(cheatId)` + emit `hotkey:dec`.

### 8.2 Semantics (decided in brainstorming Q5: A2 + B1)

```ts
// engine-host
incCheat(cheatId): { ok: true } | { error: string } {
  const cheat = supportedCheatById(cheatId);
  if (!cheat || cheat.kind !== 'value') return { error: 'not a value cheat' };
  if (!isActive(cheatId)) return { error: 'cheat not active' };
  const cur = currentValueOf(cheatId);
  const next = clamp(cur + cheat.step, cheat.min, cheat.max);
  if (next === cur) return { ok: true };               // saturated
  await setValue(cheatId, next);                       // updates freeze target (B1, free)
  emit({ type: 'cheat:value-changed', cheatId, value: next, cause: 'hotkey' });
  return { ok: true };
}
// decCheat is symmetric, subtracts step.
```

Edge cases:
- Inc/Dec on a `toggle`-kind cheat: error returned, ignored silently.
- Inc/Dec while cheat is off: silently ignored (matches Cheat Engine).
- Saturation at `min`/`max`: silent no-op.
- Same accelerator bound twice: `globalShortcut.register` returns `false`; logged and skipped.

### 8.3 Renderer

`trainer-store` already listens for `cheat:value-changed` (Phase 4 Task 9) — Inc/Dec state syncs through that existing path. The new `hotkey:inc` / `hotkey:dec` events are informational only (room for a future "+0.5" toast); 4.5 does not consume them in the UI.

### 8.4 Tests

- Unit: `engine-host.incCheat` / `decCheat` clamping, gating on `isActive`, no-op on saturation, error on toggle-kind, `cheat:value-changed` emission.
- Integration: fake `globalShortcut`, register cheat with `inc: 'F4+Up'`, simulate fire, assert `setValue` called with `currentValue + step`.

## 9. Renderer Wiring & UI

### 9.1 Stores

- `library-store.ts` — `{ games, loading, error?, scan() }`. `scan()` calls `window.starlight.scanLibrary()`.
- `process-store.ts` — `{ processes, matchedPid, refresh() }`. Subscribes to `process:list` and `process:matched` on mount; `refresh()` calls `listProcesses()`.
- `trainer-store.ts` (modified) — adds `setProcessName(names: string[])` action that calls IPC and updates `currentTrainer.game.processName`.

### 9.2 LibraryRoute

- On mount: `library-store.scan()`.
- Boxart grid: each tile shows `name`, install dir hint, "running" badge if any entry of `process-store.processes` matches the game (case-insensitive name compare against `installDir` basename — coarse heuristic for 4.5; Phase 5 catalog will provide the canonical `processName[]`).
- "has trainer" badge: stays as a placeholder, always `false` in 4.5 (Phase 5 catalog wires it up).
- Boxart URL: `https://cdn.cloudflare.steamstatic.com/steam/apps/{appId}/library_600x900.jpg`. Plain `<img>` with `onError` falling back to the existing neutral placeholder tile.
- "Refresh" button → `scan()` again. Loading state → skeleton tiles.
- Empty state: "No installed games detected. (Library auto-detection currently supports Steam.)" with a link to the existing manual `.CT` import.

### 9.3 ActiveTrainerRoute

- **Process picker** replaces the free-text PID input:
  - Combobox listing `process-store.processes` sorted by name.
  - Auto-highlight the entry whose pid matches `process-store.matchedPid`; if the user has not yet picked anything, auto-fill the value with the matched pid.
  - Free-text fallback: typing a number not in the list still works (preserves Phase 4 behavior; useful when `ps-list` misses an entry).
  - Refresh icon → `process-store.refresh()`.
- **Trainer Info disclosure** (collapsed by default): editable "Process name" field showing `currentTrainer.game.processName` joined by comma. Edit → `setProcessName()` IPC.

### 9.4 Tests

- `library-store.test.ts` — fake IPC; loading and error transitions.
- `process-store.test.ts` — fake event bus; `matchedPid` updates and dedup.
- `LibraryRoute.test.tsx` — fake stores; grid render, empty state, running-badge logic, Refresh.
- `ActiveTrainerRoute.test.tsx` (extend) — picker shows processes, auto-fill on match, manual edit of `processName` flows through IPC, free-text PID fallback still works.

## 10. Risks

- **`globalShortcut` on Linux Wayland is partial.** Phase 4 already flagged this; Inc/Dec inherits the same limitation. No change in 4.5.
- **Steam registry read on Windows** requires either a small native helper (`reg.exe` query) or a tiny dep. Plan: shell out to `reg.exe` via `child_process` — no new dep, available on every Windows install.
- **`ps-list` enumerates all OS processes every 2s.** The window-hidden / attached gates are the load-bearing optimization. If they regress, polling becomes a CPU drain on idle laptops.
- **VDF parser is hand-rolled.** Failure mode is `[]`, not crash. Fixture coverage must include malformed input.
- **Heuristic "running" badge on Library tiles** (matching by `installDir` basename) will have false negatives. Acceptable for 4.5 — Phase 5 catalog provides the canonical `processName[]`.
