# Phase 6.1 — Auto-Discovered fearlessrevolution Catalog Design

**Status:** Design (pre-plan)
**Predecessor:** Phase 6.0 (committed; Wayland hotkeys via uiohook-napi).
**Successors:** Phase 6.2+ (Linux packaging, Windows packaging, macOS signing, auto-update, polish).

## 1. Goal

Replace the hand-curated 5-entry placeholder catalog with thousands of real fearlessrevolution trainers, auto-discovered from the public phpBB forum and auto-downloaded at indexer time. End-user experience: click a Library tile → trainer auto-fetches → cheats activate. No manual download.

## 2. Non-Goals

- Scraping sources beyond fearlessrevolution. Future phases can add `discover-mrantifun.ts` etc. with the same pattern; out of scope here.
- Authenticated downloads. Empirical testing confirms `download/file.php?id=N` returns `.CT` bytes anonymously; we keep it simple.
- Catalog hand-edits surviving regeneration. `seeds.yaml` is fully generated by `discover`; the 5 hand-curated Phase 5.0 entries get replaced. A future `seeds.overrides.yaml` could merge user-curated entries on top of discovered ones, but is out of scope for v1.
- Smart trainer-version selection. We pick the last `.CT` attachment in each thread (typically newest), not the one matching the user's installed game version.
- First-run cap or progressive backoff. The initial run downloads ~5,000 attachments at 1 req/sec (~80 minutes); the SHA-256 cache from Phase 5.4 keeps subsequent runs cheap.

## 3. Architecture

Five new and modified pieces, each with one clear responsibility:

```
packages/indexer/src/
├── index.ts        (modified — adds 'discover' subcommand dispatch)
├── seeds.ts        (modified — SeedEntry adds optional rawTitle, processName defaults [])
├── fetch.ts        (modified — branches on viewtopic.php URLs)
├── slug.ts         (unchanged from Phase 5.4)
├── write.ts        (unchanged from Phase 5.4)
├── discover.ts     (NEW — phpBB forum walker)
├── title.ts        (NEW — aggressive title cleaner)
└── steam.ts        (NEW — Steam GetAppList lookup with 24h cache)

apps/desktop/src/renderer/routes/
└── LibraryRoute.tsx (modified — match catalog by name when steamAppId is null)
```

**Boundaries:**
- `discover.ts` is the only place phpBB HTML is parsed.
- `fetch.ts` gains one branch and one helper; the rest of the file (zip extraction, error handling) is unchanged.
- `title.ts` is a pile of regex helpers, fully unit-testable without a network.
- `steam.ts` does one thing — `loadSteamAppList()` returns a `Map<lowercase-name, appId>`.
- The desktop app touches one file. All other catalog plumbing (Phase 5.0 client, Phase 5.5 SteamGridDB fallback) stays unchanged.

## 4. Discovery (`discover.ts`)

### 4.1 Algorithm

```
const FORUM_BASE = 'https://fearlessrevolution.com/viewforum.php';
const FORUMS = [4];                       // cheat-tables forum
const PAGE_SIZE = 50;                     // phpBB default; 'start' increments by 50
const SLEEP_MS = 1000;                    // throttle: 1 req/sec

for forum in FORUMS:
  start = 0
  while true:
    page = fetch(`${FORUM_BASE}?f=${forum}&start=${start}`)
    threads = extractTopics(page)         // [{ id, rawTitle }]
    if threads.length === 0: break
    for thread in threads:
      if isStickyOrRequest(thread.rawTitle): continue
      cleaned = cleanTitle(thread.rawTitle)
      seeds.push({
        url: `https://fearlessrevolution.com/viewtopic.php?f=${forum}&t=${thread.id}`,
        name: cleaned,
        rawTitle: thread.rawTitle,
        steamAppId: null,                  // filled in §4.4
        processName: [],                   // desktop fills at trainer-load time
        platform: ['windows'],
      })
    sleep(SLEEP_MS)
    start += PAGE_SIZE
```

The walk terminates when phpBB returns a page with no `class="topictitle"` matches (past the last page).

### 4.2 Topic extraction

Single regex against the raw HTML:

```ts
const TOPIC_RE = /<a href="\.\/viewtopic\.php\?f=\d+&amp;t=(\d+)[^"]*" class="topictitle"[^>]*>([^<]+)<\/a>/g;
```

`class="topictitle"` filters out pagination links (`start=...`) and per-post anchors (`p=...#p...`). Captures `(id, rawTitle)`. Decoding HTML entities in rawTitle (`&amp;` → `&` etc.) is a single pass after capture.

### 4.3 Sticky / request filtering

```ts
function isStickyOrRequest(rawTitle: string): boolean {
  const t = rawTitle.toLowerCase().trim();
  return /^\[?(request|req)\]?[\s:]/i.test(t)
      || /^before you upload/i.test(t)
      || /^cheat engine download/i.test(t)
      || /^forum rules/i.test(t)
      || /^\[important\]/i.test(t);
}
```

Stickies and requests are dropped at discovery time so they never appear as catalog entries.

### 4.4 Steam ID enrichment

After the forum walk completes:

```ts
const steamMap = await loadSteamAppList();             // §6
for (seed of seeds) {
  const id = steamMap.get(seed.name.toLowerCase());
  if (id !== undefined) seed.steamAppId = id;
}
```

Misses stay `null` and get SteamGridDB-resolved boxart at render time (Phase 5.5).

### 4.5 Output

`packages/indexer/seeds.yaml` written via `atomicWrite` (Phase 5.4 helper). One YAML document with the full `games:` array. Logs:

```
Discovered N games (M with Steam IDs, K filtered as request/sticky)
```

### 4.6 CLI dispatch

`packages/indexer/src/index.ts` gains an argv check at the top of `main()`:

```ts
const subcommand = process.argv[2] ?? 'index';
if (subcommand === 'discover') {
  await discover();
  return 0;
}
// ... existing index logic
```

So users run `node packages/indexer/dist/index.js discover` to refresh seeds, then `node ...` (no arg) to ingest.

## 5. Title cleaning (`title.ts`)

Aggressive cleanup with a fallback to the raw title preserved on the seed.

```ts
export function cleanTitle(raw: string): string {
  // 1. Decode HTML entities
  let s = raw.replace(/&amp;/g, '&').replace(/&#039;/g, "'").replace(/&quot;/g, '"');

  // 2. Strip leading decorations: [REUPLOAD], [Steam], [Updated], etc.
  s = s.replace(/^\s*(\[[^\]]*\]\s*)+/g, '');

  // 3. Cut at the first separator that introduces metadata.
  //    | — table v: 1.0   ── Cheat Table   - Achievement Table   (Steam)
  const cutAt = s.search(/\s*([|—–-]\s|─{2,}|\(Steam\)|\[Steam\]|\bv\d|\bV\d|\bUpdated:|\bachievement table|\bcheat table|\btrainer\b)/i);
  if (cutAt > 0) s = s.slice(0, cutAt);

  // 4. Trim whitespace, drop trailing punctuation
  s = s.trim().replace(/[\s|:─.;,]+$/, '');

  return s.length > 0 ? s : raw.trim();
}
```

The function is intentionally regex-heavy and relies on test coverage for confidence:

```ts
describe('cleanTitle', () => {
  it('strips |-delimited version metadata', () => {
    expect(cleanTitle('Crusader Kings III | Steam v1.18.2 | Updated: 2025-12-12'))
      .toBe('Crusader Kings III');
  });
  it('strips trailing decorations like (Steam)', () => {
    expect(cleanTitle('Horny Villa (Steam) ── Cheat Table')).toBe('Horny Villa');
  });
  it('strips leading [Steam] / [REUPLOAD]', () => {
    expect(cleanTitle('[Steam] Escape Dungeon 2')).toBe('Escape Dungeon 2');
    expect(cleanTitle('Dead Rising 3 [REUPLOAD]')).toBe('Dead Rising 3');
  });
  it('strips - table v: notation', () => {
    expect(cleanTitle('R.U.S.E. - table v: 1.0 CT')).toBe('R.U.S.E.');
  });
  it('strips - Achievement Table - V<n>', () => {
    expect(cleanTitle('Europa Universalis 5 - Achievement Table - V1.0.10'))
      .toBe('Europa Universalis 5');
  });
  it('preserves colons inside game names (Need for Speed: Heat)', () => {
    expect(cleanTitle('Need for Speed: Heat')).toBe('Need for Speed: Heat');
  });
  it('returns raw input when result is empty', () => {
    expect(cleanTitle('  ')).toBe('');
    expect(cleanTitle('|||')).toBe('|||');
  });
});
```

The "preserves colons" test pins the behavior that `Title: Subtitle` survives — the regex cuts at ` - ` / `|` / `(Steam)` / version markers, NOT on `:`.

## 6. Steam lookup (`steam.ts`)

```ts
const APPLIST_URL = 'https://api.steampowered.com/ISteamApps/GetAppList/v2/';
const CACHE_TTL_MS = 24 * 60 * 60 * 1000;

interface AppEntry { appid: number; name: string }
interface AppList { applist: { apps: AppEntry[] } }

export interface LoadOpts {
  fetch?: typeof fetch;
  cachePath?: string;       // defaults to <pkg-root>/.steam-applist-cache.json
  now?: () => number;       // for testing TTL
}

export async function loadSteamAppList(opts: LoadOpts = {}): Promise<Map<string, number>>;
```

### 6.1 Cache flow

1. If `<cachePath>` exists and `now() - mtimeMs < CACHE_TTL_MS`, parse and return.
2. Otherwise: `fetchFn(APPLIST_URL)` → JSON → build `Map<lowercased-name, appid>` → atomic-write the JSON to cache → return the Map.
3. On fetch failure: return an empty Map (graceful degradation; discovery proceeds with all `steamAppId: null`).

### 6.2 Caveats accepted

- 250,000+ entries: dedicated server tools, soundtracks, demos. Some forum titles will match the wrong record (e.g., a soundtrack instead of the game). v1 accepts this — the trainer still works; only the boxart is wrong.
- Multiple entries per game (deluxe editions, regional variants). The map only retains the last appid seen per lowercase name. Non-deterministic, fine for v1.
- Linux-native and non-Steam games miss entirely. SteamGridDB fallback covers boxart.
- API is unauthenticated and CDN-cached — no API key, no rate-limit risk for one fetch per indexer run.

### 6.3 Tests (`steam.test.ts`)

```ts
it('builds a lowercased map from GetAppList JSON');
it('caches result; second call within TTL does not refetch');
it('refreshes after TTL elapses');
it('returns empty map on fetch failure');
```

The test injects a synthetic `fetch` returning a small fixture JSON, plus a controllable `now()` to simulate cache aging.

## 7. Fetch extension (`fetch.ts`)

The current `fetchTrainer(url)` from Phase 5.4 expects direct `.CT` or `.zip` URLs. Add a single branch.

```ts
const VIEWTOPIC_RE = /^https:\/\/fearlessrevolution\.com\/viewtopic\.php\?/;

export async function fetchTrainer(url: string): Promise<Buffer> {
  if (VIEWTOPIC_RE.test(url)) {
    const directUrl = await resolveViewtopicAttachment(url);
    if (!directUrl) throw new Error(`viewtopic ${url}: no .CT attachment found`);
    return fetchTrainer(directUrl);                    // tail-recurse
  }
  // ... existing direct-URL logic
}

async function resolveViewtopicAttachment(topicUrl: string): Promise<string | null> {
  const res = await fetch(topicUrl, {
    headers: { 'User-Agent': USER_AGENT, Accept: 'text/html' },
    redirect: 'follow',
    signal: AbortSignal.timeout(FETCH_TIMEOUT_MS),
  });
  if (!res.ok) return null;
  const html = await res.text();
  const matches = [...html.matchAll(
    /<a[^>]*href="\.\/(download\/file\.php\?id=\d+)"[^>]*>([^<]+\.(?:CT|ct|zip))<\/a>/g,
  )];
  if (matches.length === 0) return null;
  // Prefer .CT files; fall back to .zip. Within preferred type, last wins (newest).
  const cts = matches.filter(m => /\.(?:CT|ct)$/.test(m[2]!));
  const pick = (cts.length > 0 ? cts : matches).at(-1)!;
  return `https://fearlessrevolution.com/${pick[1]}`;
}
```

### 7.1 "Last wins" rationale

phpBB renders posts oldest-first. Threads start with the original `.CT` and accumulate revisions as new posts. The last attachment is the maintainer's most recent revision. Parsing post timestamps would be more robust but the HTML is verbose and version-dependent; ordinal position is stable and free.

### 7.2 Cookie handling

`download/file.php?id=N` works anonymously without a session cookie. Empirically verified against thread `t=13576` (Crusader Kings III) — `curl` without `-b` succeeds, returning a valid `.CT` XML file. We keep the implementation simple: no cookie jar, no `credentials: 'include'`. If a future site change requires it, we add cookie handling then.

### 7.3 Tests (extending `fetch.test.ts`)

```ts
it('viewtopic URL: prefers .CT over .zip when both present');
it('direct .CT URL: unchanged from Phase 5.4 behavior');
```

## 8. Library matching (`LibraryRoute.tsx`)

Currently the Library tab matches `hasTrainer` via `catalogSteamIds.has(steamAppId)`. With auto-discovered entries that have `steamAppId: null`, this misses.

Add a name-keyed index alongside the Steam-ID set, and a single `findCatalogEntry` helper that consolidates the previously inline lookup logic.

```ts
const catalogNameIndex = useMemo(() => {
  const m = new Map<string, CatalogIndexEntry>();
  for (const g of catalogIndex?.games ?? []) {
    m.set(g.name.toLowerCase(), g);
  }
  return m;
}, [catalogIndex]);

function findCatalogEntry(g: DetectedGame): CatalogIndexEntry | undefined {
  // Fast path: Steam-ID match wins when both sides have one.
  const steamId = g.boxartSteamAppId ?? (g.source === 'steam' ? Number(g.appId) : NaN);
  if (!Number.isNaN(steamId)) {
    const byId = (catalogIndex?.games ?? []).find(e => e.steamAppId === steamId);
    if (byId) return byId;
  }
  return catalogNameIndex.get(g.name.toLowerCase());
}
```

The render block uses the result for both the badge and the click handler, so each tile gets a single match decision:

```tsx
{games.map((g) => {
  const matched = findCatalogEntry(g);
  return (
    <GameTile
      key={`${g.source}:${g.appId}`}
      game={g}
      running={runningSet.has(dirName)}
      hasTrainer={!!matched}
      onClick={matched ? () => void openTrainerForEntry(matched) : undefined}
      {...(g.source === 'manual' ? { onRemove: () => void removeManual(g.appId) } : {})}
    />
  );
})}
```

`openTrainerForGame` becomes `openTrainerForEntry(entry)` — fetches the trainer file, calls `setActiveTrainerFromCatalog`, navigates to `/active`.

## 9. Etiquette

The discoverer makes ~100 forum-listing requests (~5,000 threads / 50 per page) plus the indexer's per-seed downloads (~5,000). Total ~5,100 requests on a cold first run, ~100 every weekly cycle thereafter (forum walk + delta only).

- **Throttle:** 1 req/sec (`SLEEP_MS = 1000`) for both the forum walk and the per-seed download loop.
- **User-Agent:** `starlight-indexer/0.0 (+https://github.com/darkharasho/starlight)`. Identifiable; lets the site operator contact us if they want us to back off.
- **robots.txt:** check at startup; abort the run with a warning if `Disallow: /viewforum.php` appears. fearlessrevolution's current `robots.txt` allows it.
- **Cache-respecting:** the SHA-256 cache from Phase 5.4 means we don't re-download unchanged attachments week-over-week — most weekly runs make ~100 head-only requests + 5–20 actual downloads.

## 10. Risks

- **Forum HTML changes.** If fearlessrevolution updates its phpBB theme, the topic-extraction regex could break. The discoverer's failure mode is "discovers zero topics → keeps existing seeds.yaml unchanged" (we should write the new seeds.yaml only on non-empty results — see §11).
- **Title-cleaning false positives.** A game named "Need for Speed: Hot Pursuit" includes "Pursuit" and survives cleanup, but a game named "Need for Speed - 2015" loses its year. Acceptable; `rawTitle` is preserved for rare manual disambiguation.
- **Steam-ID misalignment.** Lowercased exact match against GetAppList means "Stardew Valley" matches the game, but "Stardew Valley Soundtrack" matches the soundtrack. When we pick the soundtrack's appid, the Library tile shows the soundtrack's boxart. v1 accepts this.
- **Indexer first-run time.** ~5,000 fetches × 1 sec = ~80 minutes. The cron runs in CI on `ubuntu-latest` (no time limit per job within reason). Deltas are fast.
- **Forum download cookie requirement creep.** Empirical test confirms anonymous downloads work today; site operators may add auth in the future. Mitigation: when the indexer hits a 401/403 on `download/file.php`, log + skip the seed; the existing per-seed isolation prevents one bad seed from breaking the run.
- **Catalog churn vs. existing pinned trainer.** A user who has loaded a trainer from the old catalog (id `elden-ring`) and the indexer regenerates the catalog with a different `id` for the same game (e.g., `elden-ring-2` from a newer thread) means the user's `recents` entry points to a missing id. Phase 5.1's recents render handles missing-catalog-entry gracefully (renders non-clickable). Acceptable.

## 11. Failure modes

- **Discoverer hits a network error mid-walk.** The walk fails closed: throw out of `discover()`, do NOT overwrite `seeds.yaml`. The next run resumes.
- **Discoverer produces zero seeds.** Refuse to write `seeds.yaml` (preserves the existing file). Log a warning. The HTML structure may have changed.
- **Indexer fetch fails on a viewtopic URL.** Per-seed try/catch already isolates failures (Phase 5.4); failed seeds fall back to their previous `index.json` entry (Phase 5.4 review fix). Catalog stays stable.
- **`steam.ts` fetch fails.** Returns empty map; all seeds get `steamAppId: null`. SteamGridDB fallback handles boxart.
